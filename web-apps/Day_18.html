Javascript Introduction

I will not lie, I did not expect us to ever learn JS in this web apps class.

The first ~15 minutes he talks about the exam and misc. things. At 14:25 he switches over to his JavaScript lecture.

Due to our professor's newly found video editing skills, we have skipped time into the future. The artificial light of yesterday is no more. We are met with natural light and my greatest enemy written in somewhat legible text: JavaScript. Have mercy...

He flips the giant pages back while remarking how he wishes he had a whiteboard to work with.

Our professor could not have written truer words: JavaScript strikes me as a rather sloppy, hodgepodge, of adhoc facilities. Things that have been just tacked on willnilly. It resembles throwing a pile of fruit into a blender that doesn't work very well.

I appreciate this analogy.

JavaScript was originally invented in an attempt to take over the web as other people were trying to use the Java language. Somewhere in this mess, we got flash which was also an attempt to take over the web (flash thought they could ask for royalties and boy that sure worked out. Note, it did not work out. RIP old newgrounds.)

Instead of Java or Flash, we ended up with JavaScript which got its name in a deliberate attempt to confuse the user into believing it had something to do with Java. It's sorta object oriented like Java so there's that.

Anyways, we shall be referring to JavaScript with one of its many names: ecmascript. Why? Because that was the name given when people tried to get certification for JavaScript.

Now then, JavaScript has many problems which Microsoft tried to solve with TypeScript which adds type checking to JS.

Gross says he'd like to give us a book like he did with Scheme but he hasn't found one. He says to just check out something like w3schools.com (sigh. aw shit here we go again). He likes the w3schools website but admits it's quite fragmented to which I highly agree.

JS is meant for the web but nowadays it's becoming popular to use it off the web too (Node.js anyone?). He's going to limit our discussion for web pages though.

He is going to give us a template for a typical web page using JS: 

<html>
<head>
	functions
</head>
<body>
	functions
</body>
</html>

There are many tags but a simple html tag will do. We also have the head tag which we can have the title and all sorts of other metadata. The most important thing you can put in the head is JS functions. Function definitions in the head is good practice. Very helpful for debugging a program. You'll know exactly where to look for them. The head tag is following by the body tag. (The end </body> tag isn't technically necessary. Browsers are smart enough to know where it ends.)

You can also put your functions are the bottom of the body tag as well. However, if you have a lot of code your browser is going to take a while to get to the code. It's gonna take a while before your browser can display anything and that is a problem.

Back with time sharing stuff, people asked how long would a user wait for something to load? It was about 5 seconds. Any longer and users get pissed off. If you have a lot of code, it pays to stick your functions at the very bottom of the HTML. That way, the user will have something to look at while the browser interprets the code.

So, stick function definitions in the head or stick them in the bottom of the body. You could stick your code wherever but there are various issues that come with this. It's not good practice to just stick your JS code wherever.

Now that we have the basic template of HTML code, let us move on to the script tag.

How does the browser know that it's dealing with JS? It uses the script tag! In between the script tag we put JS. When the script tag closes, it's back to regular HTML. 

<script>
	javascript
</script>

For a time, browsers didn't support JavaScript. In fact, some users just disable it! There is a noscript tag we can use to detect if the browser is down to do JS. We can then provide alternative text. Sadly, the code will then display. He's not gonna go into any of this though.

Now then, let us go into JS itself. One of the most important facility in JS is comments. There is // and multiline /* */ which we're used to. 

// for single line comments

/*
Multi-
line
comments
*/

Remember, if it aint documented it aint! Gross says he did see a documentation issue that is of interest. There is an important simulation of the COVID-19 virus. That model needs some modification. The updating has been seriously delayed because it consists of 1000s of lines of undocumented C code. The updating is delayed now as someone documents it. This is a bloody mess.

As mentors back in Gross' time recommends, it's often easier to rewrite code then figure out how undocumented code works (I feel that professor).

Given the rate that COVID-19 is moving, speed is of great concern. Document heavily!

Now then, let us begin looking at JS proper!

As usual, one of the first things we need is a place to put the data that we are going to work with. It needs to be stored internally in memory. As with any imperative language, we use variables. No matter the language, variables need to be declared somehow so that our interpreter/compiler knows. The type of variable, how much space is needed for it, etc. In C++ and Java, we have excplicit declarations. A datatype, a name, and perhaps an initialization. JS gives us something like that but it is very weakly typed and dynamic typed. Variables are given a type on the fly! That type can be changed on the fly too.

Assigning a type and compelling an interpreter to keep that type can be a nuisance. The stronger the typing the stronger the enforcement of the rules. Physicists do the same when making calculations for if the resulting type of data is wrong then they know something went awry.

In any case, we have two ways to declare JS variables: 

var myvar = 42; // we can use the var keyword
myvar = 3;      // or we could just straight up initialize it like so
myvar = "text"; // we just changed the type! we can do this too
a = b + c;      // or we can just use it in an expression. We just declared a!

Despite all the different ways of declaring and initializing variables, it's bad practice to just declare+initialize all willnilly and wherever. Declare it first at the top.

Next, we have functions. We are used to declaring functions with a return type, function name, and then an argument list. JS is terribly careless about typing. So, the type of return data is merely a matter of what you actually return. So we don't define what the return type is.

We do make use of a function keyword followed by a function name, an argument list, and the body of the function. 

function funcName() {
	// you can declare local variables
	// when we return, they go away
	// we don't really have static variables in JS
	// we instead have global variables.
	// You can declare variables outside any function
	// and they remain indefinitely (messy. Very messy)
	var yourVar;
	return yourVar;
}

// There are a few important functions. Perhaps one of the most important
// is the alert box. The function looks like so:
alert(myString); // takes a single string. It can be multiple lines with \n

// alert is nice for debugging albeit a bit crude

We can also debug JS with the web page itself such as making changes to the HTML. The problem is that we have no idea who is doing what. You'll have to dig around but, nonetheless, it is very useful for output. 

<p id="demo">text</p>
<script>
	// We can change the HTML just like so:
	document.getElementById("demo").innerHTML = "new text";
</script>
<!-- We can also include external JS -->
<script src="source.js"></script>

When the interpreter gets to line 4 in the above code, it'll change the HTML in the page.

To make these changes, we need to get our grubby wittle paws on something in the page uwu. Use getElementById().

In the words of Gross, JS is a relatively easy language to learn so long as you don't pay attention to details and good programming practice. lmao nice.
