Day 20

Created Tuesday 28 April 2020

Video can be seen here.

He begins the video by telling us how to put our game online. I just used my own home server instead.

Now he's talking about declarative vs imperative languages. Don't deal with declarative languages as though they are imperative! Imperative languages has us translate some value to some representation inside of the machine. This is what the assignment statements are all about. You have to store the values to operate on them! In imperative on the other hand, we deal with those values directly. Hence the whole aliasing thing we do with define in scheme.

Internal representations are why we need pointers.

In imperative, they're recipes that we follow and hope they give us the correct result.
In imperative, we don't care how it gets the sum. Just tell us the answer!

In the CADET, there was no arithmetic! Instead, you used a table lookup. (can't add, doesn't even try).

Procedural languages like C or Java, functions are precedures that we follow. Could be in ram or cache or wahtever.

In declarative, we try to avoid the machine all together. No worrying about allocating memory and whatnot. We don't want to bother with all the extra stuff. Just get it done. In fact, there aren't really any pointers in declarative languages either!

He references an article called, "Computing Without Data" which started quite the storm. This was before declarative was really starting to pickup.

A little lazyiness is a good thing for a programmer.

Let us look at scheme again: 

(define a 2)

a is a symbol and 2 is a value. The scheme function define just connects them. There is no modification here. Whenever you see the symbol a, just replace it with 2.

He notes many people thought define made a into a function. They were getting confused with the following though: 

(define a (lambda (n) (rule)))

Chances are, the reason for this mixup is because they were using the short form to declare a shceme function that doesn't use lambda. 

(define (a x) (rule))

The above should translate to the lambda version. As such, people got confused.

He notes that HTML is a remarkably declarative language! <b>text</b> says that the text between the tags should be bold. The bolding is left up to the browser.

HTML and CSS are declarative languages he notes!

Make sense. We declare what we want to happen and then it does it without us worrying about the details.

He made the oracle joke again. I think this was upwards the 10th time.

map function

Now he's going over the map function in scheme. 

(map function list)

Let's say we have the list (0 1 2 3 4 5) and we want to square each one.

It would be a real bother to define a whole squaring function so we can do it with just lambda! 

(map (lambda (x)(+ x x))
	(0 1 2 3 4 5))

Very useful especially when we only need a function once. There's no need in naming it afterall.

Lambda functions were invented because mathematicians realized you really don't need a name. It requires a level of indirection that we'd like to get rid of. Functions that aren't burdoned by names.

What is a function in a declarative language?

  1. It is a set of arguments that will be replaced by parameters.
  2. Together with a rule for combining the parameters (or args) into a simpler expression.
    1. Simpler than the rule itself he notes.

We'd like to have all the arguments replaced by parameters but we don't have to do that. Functions are merely a sort of value. We don't have to reduce them to a collection of operations. We can think of them as a symbol! A function is merely a symbol just like all the other symbols we work with (a being a symbol for 2 for example).

Functions are first class values because they can be passed around. We can pass functions. They're merely symbols.

Functional programming calls for us to treat our program as a transformation of our input data into output data.

IBMs HIPO defines programs strictly in terms of Input, Process, and then Output.
You can then refine each of these levels. Successive Refinement.
This helps us to organize our program.
You can write pretty good functional programming in C, C++, and Java he notes.

Variables can be given values only once. This helps with debugging and proving our programs (although only a few programs can be truly proved).
