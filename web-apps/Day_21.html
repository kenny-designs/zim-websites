More on the Map Function

Professor is back at it again with the white sketchpad.

He says to buckle up cause we'll be deep diving into the map function and a few other things as well.
He says the map function has some very interesting applications in modern computing.

We've run out of computing power he says. We can't solve modern computing problems with the single CPU machines of not that long ago.

As such, we've had to move to parallel computing. Hardly anyone uses a single processor now. It's all 2 or more at this point! Even GPUs he has in his lab have over a 1000 processors each.

In the future, we're gonna need more core as problems get more difficult. For example, modeling explosions. The volume to be affected by the explosion is split up into many small cubes. Then calculations are performed for each of these cubes. It doesn't take a lot of cubes to create a rather horrendous computation. We need all those multiple processors.

There is a strong correlation between how you should look at the map function and parallel computing.

Let's take a step back: what's a list?
A list is a set of elements. As in scheme: (1 2 3 4 5)
There is an important property though: the list is ordered!
Each item in the list has a position. 3 comes after 2 but before the 4 for example. This allows us to have a shorthand for locating items within the list. We can assign an integer to each item in the list (yknow, the index?). We then use that integer to help locate/identify each number in the list.

Having this index makes life pretty quick. You want the number 3? Go to list index 2 (remember, we start from 0).

We identify each item in an array with an index. (I believe he might be trying to make a point that an array and a list are two different things. Might be useful on the final.)

There is a difference!
Inside the computer, all of the elements in an array have to be next to eachother! Yknow, contiguous?
For a list, they don't have to be.

To locate an item in a list, we must use a procedure. For an array, just use the index and multiply by the size of each element in the array.

In a decarative language, we don't care about everything in memory. From our viewpoint, lists and arrays are the same.

What does this all have to do with map?

Well, remember map looks like this: 

(map function list)

We apply the function to all of the elements in this list and give us back a list with the results.
If we gave (1 2 3) and had each multiply by itself, then we'd get (1 4 9). However! Noticed how he never specified the order in which this is all done. This is crucial.

All of those elements in the list are subjected to the function at the same time. In an imperative language, we'd have to loop over the elements and keep track of everything with, say, having an int i as we do in for loops. We can only calculate one cell at a time though!

Can we get rid of that index?
The answer is yes! This is an advantage of working with the recursive approach to iteration.
What happens if we have more than one processor? Can we do a bunch of processes in parallel? Yes we can!

Break up our input and spread them about the processors to grind away at.

One problem we have with the for loop is dealing with interrupts while we're trying to keep track of where we are (the int i). We could parallelize if we know the index will be referred to in an interrupt. However, it is impossible to do this! We have to assume it will be referred to in an interrupt and generate code accordingly.

So, if we're working with a list how do we know where we left off? In a crude way, we could just start from the beginning again. Problem is, this is a lot of work!

This all brings us back to our friend map.

Map is a parallel instruction from our viewpoint as a programmer. Map applies it's operations to all of the elements in the input list at the same time.

Scheme does have a for structure but it's used to create lists. He recommends we don't use it.

It takes a very long time to do things like processing videos. As such, we need a way to do things in parallel.

NP-complete problems are ones in which the only known solutions are brute force. We have to try all possibilities in order to find a solution. It can be pretty costly though in the amount of time it'll take to figure it all out.

The only time you can communicate between threads is via global variables! This can be a bit of a problem in trying to keep threads from being confused about the values of variables.

Working with imperative, it sucks to have to worry about what's going on inside the machine.

The only prob with decalarative languages is having to trust the runtime system. However, they're quite good.
