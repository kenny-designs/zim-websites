Lesson number 1, global variables are poision. They do pollute the global scope albeit that's not the only reason. Java makes use of them all the time. (Also remember, we have access specifiers public, protected, and private. Also default.)

What is an object? It's an instance of a class but this does not mean what you think it is. It is a member of the class of abstract data types. However, that's not what it is. An object is a block of data. It is what is built given the specifications of a class. It has member variables and methods.

Now, why is the phrase, "an object is an instance of a class" wrong? Well, look at this: 

class myclass {
  int a;
}

class myclass2 {
  int b;
}

The above two are both instances of a class. They aren't objects. They are an instance of a member of the class.

An object is an instance of a member of a class.

An object is built given from the definitons of a class.

An object is a block of data together with methods used for manipulating that data.

Global Variables

They can be poison but there are places we must used them. However, this brings us back to the question just why they are poison? Well let's look at the following: 

myclass class {
  int a;
  void method() {
    a = 3;
  }
  
  void method2() {
    a = 3;
  }
}

a = 3 but it should be 5. What now? 

myclass class {
  int a;
  void method() {
    a = 3;
  }
  a = 5;
  
  void method2() {
    a = 3;
  }
  a = 5;
}

The problem is which method set a to 3? In fact, which method was called? Where was it called? We don't know! This is a huge problem. You have to use global variables in objects but it's gonna be hard to debug. As such, it's better to have them local if you can. This is one reason why we use accessor routines to figure out just who's modifying that variable and giving it the wrong value. Limit the access to variables. Or the modern buzzword encapsulate.

What is a function? It is a set of arguments with a rule for manipulating those arguments.

It has been the programmers job to do whatever the systems analyst told them to. Convert instructions from common language to machine code. The systems analyst was well paid and the programmers were alright. They didn't need much schooling either. Over time, there was a shortage of coders and systems analysts started writing their own code. This was a mistake. Today's classes combine problem solving with coding. This is a bit of a problem. You have to pay attention to controlling a machine while solving a problem. This is difficult to juggle. With an imperative language, you are constantly being distracted by little details such as allocating memory. What does that have to do with your problem? It's a distraction! What is important is to separate the two if you can.

This was the point of the orange pyramid problem. You'd think you just need to solve it recursively but then suddenly it gets a little complicated. Suddenly, you need to step away from the code and focus on the problem itself.
