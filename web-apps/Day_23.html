Review

Apparently he's going to try and review everything from the beginning of the class.

A computer program is not merely a set of instructions to be followed/executed by a glorified calculator. A computer program is actually the statement of the solution to a problem in computational terms.

We use computers to solve problems that require computations for solutions.

  • Solve the problem first.
  • Then restate your solution in a computer readable form.

By solving the problem first, you can concentrate on solving the problems. Vice versa, with the problem solved we can just focus on programming the machine.

Remember these three paradigms of programming:

  • Declarative - what it is we want without writing all those statements ourself. We don't care how it does it just get it done.
  • Object-Orientation (OOP) - a combination of declarative and imperative programming. We build an object and also provide a set of methods to manipulate that object. The methods are imperatives.
  • Imperative - each statement is an instruction for the computer to do something.

Functional programming (most frequently appears in declarative form) we treat a program as a functional transformation of input data into output data. However, we can do these kinds of things in both imperative and declarative programs so he doesn't consider functional programming to be its own paradigm.

If you look closely at Java and C++, you'll find that they're actually imperative languages with object extensions. You can write object oriented code in any programming language! It's easier in some languages such as Java and harder in others like prolog.

What is an object??? It is a block of data together with methods to manipulate that data.
It is not an object is an instance of a class as we often hear. A class is a specification. An object is something that is built according to a specification.

He states a big reason why we got over these three paradigms of programming is so that we know what they look like and how to apply them in big applications. Depending on the problem, one paradigm may be better than another.

In scheme, lambda returns functions that don't have names! This is why we can combine define and lambda as we do. define provides a name for the function from lambda.

When dealing with very large teams of thousands of people, we need a way to keep all the pieces together. Therefore, we need a version control system! Or VCS for short.

Subversion he notes does a good job of teaching us the principles of version control.

import brings in a whole group of files.
add puts a file under subversion control.

With checkout and commit, subversion will move and retrieve entire directories.
The update subcommand makes sure that our directory of code is consistent with everybody elses code.

Export rips off all of the versioning information.

VCS also help us to maintain numerous version of a program.
Why keep multiple versions? Well, submit early, and submit often! It a common mantra in the field. You want to keep a complete history of development. This is great for regression testing and finding bugs.

Also, version control does not backup our work! It's not a tool to just backup our work. We still need to backup the version control files too.

Once we have a bunch of components, we have to be able to put them together. We also need to make sure all versions match (are up to date, whatever that means he notes).

To do all this, we use make. Maven and cmake tried to replace it but they don't.

We have a make rule.

If we want to construct target.o, we will have to use myfile.c and myfile.h. Now makefile, I want you to make sure that these pieces as they exist today are the correct ones for the construction of target.o. And by the way! If target.o was already constructed with these file, don't do it again! Don't bother to waste the effort. How do we do this? By looking at time stamps! If target.o is older than any of the dependencies, then execute the recipe!

target.o : myfile.c myfile.h

gcc -c myfile.c -o target.o

Also, remember phony targets! What this does is that if make can't possibly construct the target, go ahead and execute the recipe. We do this very frequently for cleaning up our files

clean :

rm *

clean is not a keyword! We're just using it as a label. We could have named it anything.

So then, we use subversion to maintain versions and make to build things properly when we need them. What else are we going to use?

It would be nice if we could get some assistance in debugging. For this, we can use an IDE (he notes that he avoids them when he can. Same professor. vim 4 lyfe! emacs is okay too. intellij more like... dumb...).

He says a while back he mentioned a debugging tool called ddt (or its predecessor odt where everything had to be written in octal). In ddt, everything had to be written in hex. Another one is called gdb (which stands for g debug). Has a nice GUI to go with it he mentions. There's also ddd.

One of the most common tactics with debuggers is executing our code up to a given points and then stop and take a look around. We call these breakpoints. You can exam the contents of registers, variables, etc.

Another debugging tool is called profilers. These will give you a map of your resource usage such as how much memory are you using.

One issue with all these debugging tools is that program optimizers will eleminate unneeded variables from the final code. If you ask to see the content of a particular variable, the optimizer will have gotten rid of it. To get around this, turn off optimization.

Another problem back in the day was wasting resources by having expensive CPUs sit idle while, waiting for printers. This led to the invention of interrupts! Interrupts are a hardware compelled subroutine call.

In a language like Java, everything executed by a virtual machine (JVM). In that case, how do we handle what we'd like to be interrupts? The answer is exceptions. Interrupts don't handle exceptions. Exceptions are software simulated interrupts!!!!!!!

For interrupt handling, you have to first save the state of the machine and then restore it. Such as registers, flags, etc. There are various ways you can save machine state. Gotta refer to a manual for that.

A process has it's own memory space. Without the help of the operating system, various processes cannot access memory in other process memory states.

Threads in a given process share memory spaces though!!!!! Communication between threads is one of the places where you need to use global variables. There's no way to avoid this but there are ways to protect it.

Graphic Displays. This is another topic we went into. The xwindow system is pretty much a model for all graphics displays of today. The problem was trying to have a cheap display to display results from an expensive, high power processor. It is this problem that leads to listeners! The keyboard and mouse in the display unit are not directly connected to the program that is going to use data from them. In fact they can't be connected because they are remote! Therefore, we need special code in the display unit (the server) and the high power processing program (client) to pass data from the keyboard and mouse to the high power processing program (CPU). This code will be different from the code used if the keyboard and mouse was directly connected to the high power processing program

Web programming is all about communication.

HTML was meant to be very simple. However, people wanted more control. As such, we got CSS. People also wanted a way to run programs on the client side. As such, we got JavaScript. JS is an imperative language and fairly simple. It has numerous flaws though. It relies quite heavily on global variables. Also, it is a dynamically typed language (types are assigned to variables on the fly and can be changed on the fly). This is very error prone.

Separate your function definitions from the rest of your code. Don't put your javascript right in the middle of the body of the HTML.

Another key issue is proper selection of fonts! Choose the right font and make sure it's readable. There is also the issue of spacing between letters. Back in the day, we had letters on the metal blocks. To get the letters closer together, you'd have to literally cut the metal! Computers makes it very easy to adjust spacing of our fonts. Even still, we run into issues like with the Arial font. The letters r and n look like an m for examle. rn. Pick fonts for readability!

Also, be careful of colors! Don't put too many colors on a page and dazzle the viewer. We need to communicate information not impress people with design skills.

Also, be careful with videos! People expect too much from videos. They have time restraints, can lack detail, you have to be aware of you viewers attention span, and be aware of what you're actually leaving out. Details that may be very, very important. Don't just try to look pretty show off how clever you are. The best videos may be those of actual candid lectures.

BNF (backus-naur form). People got together in the late 1950s to create a language for communicating algorithms. The purpose wasn't to create a computer programming language, That was a sort of an addon. Algol-60. DO NOT FORGET THAT L! SPELL IT RIGHT! It has no I/O capability at all. However, they wanted a formal way to specify the syntax of a language. It was used to help make xml for example.
