Going back to the introduction of the IBM 360, they wanted a series of machines that would allow you to pick up your software from a smaller machine and run it without change to a larger machine. This became a very popular approach for other manufacturers such as GE.

IBM started this idea to have a series of machines that would have software that was compatible all the way up and down the line. The executable code for one machine could be mounted on a larger machine and it'll work. This wouldn't work with some software but these weren't typical applications.

This all gave IBM a difficult problem. They wanted an advanced OS that could run on all of the machines from the smallest ones, such as the model 30 with its 32kb of storage, all the way up to the model 95 which could handle MB. So, how do you do this? The model 30 only has 32kb and you still need some extra space afterwards.

They did allow for some differences for the machines. They figured they'd have three versions of the OS. One for small machines like the 30, another for medium machines, and one for bigger machines that took up 256kb.

The devs knew they'd need some sort of virtual memory. So, they settled to take two 4kb areas and modularize the system. You can write routines that are 4k bytes long and swap them into these two areas as required. Do you know what this did to the discs?

What happens if you're running several programs that all want to make use of the same facility? We have two choices here: We could give each user their own copy of the code. Or they could all use it at the same time!

Reentrant code: The developers said no more! What if we wrote the code so that it never, ever changes? If it never changes, then lots of people can use it at the same time right? Reentrant code is code that does not modify itself! As long as the user provides any storage that can be modified, many people can use it at the same time.

The original code doesn't change, only data pointers change. System code doesn't change.
