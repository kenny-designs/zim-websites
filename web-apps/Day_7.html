Created Tuesday 18 February 2020

Remember, imperative vs declarative. Imperative means we dictate every little step. We program the how. Declarative is only concerned with the what. You tell it to do something and it just does it. You state what you want and let somebody else handle it.

Our professor saw that some of us used the set function in the homework. He does not like this. One of the powerful things about functional programming is that the value of the function is always the same whenever we give it the same parameters. This is known as referencial transparency. We know exactly what we're gonna get everytime we invoke a function. This is not true if we use set. Set allows us to mutate the value of variables. This makes it so we don't get the same value when providing the same parameters. This is poison! Observe the following function: 

// everytime we invoke this function, we're gonna get a different value i.e. myfunct(2)
int myfunc(int in){
	static int var;
	return (in + var);
}

Using set gives us this problem. People use set because they want scheme to behave like an imperative language over declarative. We technically can have scheme act like an imperative language but our professor doesn't want us to even know how to do this. In fact, there are optimization problems with imperative. We need to do computing in parallel. We can't get by with just one core. We use at least two. Many phones come with at least four. Heck, the threadripper makes use of 64 cores. As with a GPU, we can't make it by with just a single core.

Make

Over time, many people have tried to replace make. They have failed. Why did they try though? It can be a horror.

Even still, make is relatively simple. (What is a resident routine? Look this up.)

Compiling code can take a while. Back in the day, compiling the linux kernel could take 45 minutes. Nowadays, 4-5 minutes or so. Even still, it's still useful to control the entire process. Make is good for this. Compile and combine small pieces into a finished structure. Make may realize we don't need to recompile something cause the .obj file is still good. Why waste the effort? This speeds things up considerably.

The thing is, make can execute any command line program. Anything that combines small pieces into large ones. It can execute the copy command, mv, etc. It can be used to install large systems. It has the ability to make decisions such as if/else. If linux, do this. Else mac do that. If windows,crash. I was the only one who laughed lmao.

Usually people find somebody else's make file and then make it do what they want. For hw, it seems like he'll give us one to work from. He'll give us two separate files, compile them, and put it together to form an executable. For Java, make a jar. C++ people create an exe.

For make, we have a makefile. It has some simple commands.

target: myprog.o (or .exe. Java people use class or whatever) 

target: dependencies (files that are needed to construct the target)

If any of the dependencies are missing, we can't construct target! If one of the dependencies or whatever has changed, we need to reconstruct it. But how do we know if one of them is newer than target? There's a simple way to find out. Just check the date of when it was last modified. Every date has a time stamp of when it was last modified. Last saved. If the time stamps of target and the dependencies are different, we know if there is a change. We can check the time and date. (Yay I got that one right).

Now then, how do we find the recipe? As per legacy code, we use the tab. You know, \t. Followed by a list of programs to be executed. So, like this: 

target: dependencies
	programs to be executed. This is our recipe

The previous is the easy part. Caution! It is the only easy part. It gets insane :\

You do not want to ever accept a make default if you can avoid it. It's gonna take some care... You don't want to let a blender pick the speed for example. The blender speed is always wrong. Make defaults aren't always wrong but it can be.

We can even substitute strings which is nice.

There are also some other gimmics. One of which is called a phony target. This is a target that has no dependencies! How can we construct it if there's nothing to build it from? Make knows though. It knows to just DO IT! It has no dependencies so just execute the recipe without further question! 

phony:
	DO IT!

There's also clean. It gets rid of all the junk that's left over such as a bunch of object or class files (C++ and Java respectively). We want everything cleaned up. We make it a phony target which means only the recipe is executed. 

clean:
	takes out the garbage

Warning, the use of clean is so common that people think it's a keyword. IT ISN'T! CLEAN IS A LABEL THAT WE JUST SO HAPPEN TO COMMONLY USE! WE COULD HAVE CALLED IT GEORGE OR SOMETHING. DO NOT BELIEVE 'CLEAN' IS A KEYWORD!

Our professor also wants to make it clear that we will be using make at some point. We need to know of how a makefile works. At some point, we may even have to use its attempts at replacing it such as... maven... :'( I hate maven so much.

Remember, make helps to reduce compile time on large projects. We can use it to control which version of something we have. We could even use it to maintain databases if we're careful about it. So long as we have a large process made out of small pieces, make is great at controlling that process.

IBM's JCL is the only thing that is close to replacing make. It could execute anything and make decisions. However, it couldn't work off of timestamps...

Make is used heavily such as for installation or controlling software construction of larger programs. Heck, it's even good for smaller programs to make sure it's correctly constructed to avoid typing.

By the way! The makefile doesn't have to be named makefile. There's a command that lets you call it something else. However, makefile is the default name and you don't have to type it on the command line. So, it's just easier to leave it at the default.
