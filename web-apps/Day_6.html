CYA - Cover Your Ass

Document everything. Use version control to keep a running log. (Google docs isn't a good way to keep documentation. Can't sort by date. It's up to QA to document everything.)

By the way, cool fact: All of the best 500 super computers in the world run on linux.

Now then, how do we construct large structures from small ones? It starts with source code. The next step is to hand it to a compiler (machines can't execute human readable code afterall. It needs it in machine language). What do you get from a compiler? Not assembly code. Let's so that was already done. You get machine code as a .obj file. An object file.

Now then, what is printf? Not what it does. Now then, the compiler doesn't know. What happens now is a linkage editor (there are other terms but our professor prefers this one). It links the object files to create an executable program. A .exe. To do this, the linker goes to search a variety of libraries and combines the .objs into a .exe file. The .exe has most of the code that's going to be executed but not all of it. Why not all of it? That's because some programs are used so frequently that they are resident. There's no need to include it in the .exe. As such, we simply leave a warning that the connection is going to have to be made.

Next, after the .exe, we have a loader. A ld. This fixes up addresses to account for holes. The loaders connects us to resident programs/code. It connects them all together. And now we have something that can be executed!

What we need to look at is a program that controls the course code, obj, and linking (or mabe he pointed to the exe. close enough). We could do it manually but it's difficult. As such, we make use of a program called make. The IDE does not make use of it. It uses some version of it such as maven but it's doing the job that make does. We'll be looking more into this next week.

Homework

As you work on a program, have you ever noticed that you keep copies of your code just in case you might mess everything up. People noticed this so they used RCS - revision control system. Keeps a generation number to a file. Everytime you save a file, it increments the generation number. However, it takes up a bunch of space! And this was during a time where space was extra limited. Not to forget to mention that programs could get very large. We had no indexing either. It got pretty messy.

As a result, we invented CVS. It collected all files dealing with a project in one place. A repo. Second, it did a hell of a job squeezing files. Base changes + delta -> current version. Given the power of our machines, this worked well. However, CVS makes you deal with your project file by file. This can be painful when you have a ton of files or only want the ones in one particular directory. CVS is still used in some places but some folk got together and said let's do it directory by directory. You likely don't just want a single file but everything within the directory. Disk space is cheap now so who cares?

Subversion. Subversion works on directories. You could force it to work on single files such as with remote services. (btw, our homework will be based on subversion. Use the CLI). Now then, Linux Torvalds wasn't having a good time with subversion. Working remotely was difficult. All sorts of conflicts arose when trying to merge things. He tried the propriety bitkeeper but didn't like it. As such, he invented good 'ol GIT.

As of now, subversion and git are the two primary version control systems that we use. Note, these two aren't ran so well in the field. GUI and commercial services are trash. Don't use them. Do you know how often github backups? What if it had a head crash? You know about your directories but what about others? Are they important! Damn right they are! Commercial services will keep this info from you. You're gonna need to know what the individual directories of subversion are.

MySpace, for example, didn't back up it's data at all. 13 years of data now gone!

Now then, to the homework. We'll be using subversion. It's a little harder than git. However, subversion is prefered when you have good communication. Everyone sees eachother everyday, constant talking etc. Git is good for when communication or flaky or weak satellite links. There do exist some commercial version control systems. Some good some terrible.

Homework for Specs

You need to find the IEEE website for specifications. You don't need to do every little thing. A fragment is fine and don't pay the IEEE.

Specs is a little weird. It works sorta in bits and pieces. Before you start a project, you'd usually talk to the stake holders. They're idiots. Paper pushers. As such, who knows what's actually required? Who knows what they need to have? The answer: the end user. The best info you're going to get about the functioning of a program is the end user. For this semester's game, the people we are making it for is us! We're the end users. However, in the real world the end user won't know all that technical stuff. Keep it simple. As in, this is what the screen should look like! That's a perfectly reasonable way to do things. Heck, even a screenshot will do. Make it real easy to understand what the code is about.

Step 1: Talk to the end user.
Step 2: More is better. The more detail the specs contain the better.

Imagine this scenario, you need nuts and bolts. You order them. Takes a day or two. You get them and send them to where they need to be. But oh no! Turns out, the dimensions were wrong. The spec was incorrect.
