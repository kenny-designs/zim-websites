Day 2

1/30/2020

Imperative - how to get something
Declarative - what we want

Notice how the focus of the following code is us telling the machine what to do. Like following a recipe for a cake, we don't know what cake we'll get. We just do this and hope we get what we want. 

c = a + b;

The scheme equivalent is: 

(define c (a+b))

Here, we don't care about the details of how this is done. We don't care if there's a lookup table or whatever. All we care about is that c is defined. Where are we gonna store it? Not our concern. c is merely an alias for, say, the number 3. There's no machine or memory. No pointers. In Java, it has no pointers because it has references. In scheme, it has no pointers because it has no memory.

You see, a, b, and c for our Java and C/C++ languages are actually pointers.

In our version of scheme, it is limited. As seen back in Introduction, we only have one operation. Evaluate. And it looks like this: 

()

That's it. Just parenthesis.

Scheme is rather liberal with naming. We can have a variable named something like eg?.

The syntax for scheme is merely a function name followed by args. Every single time. 

(func arg1 arg2)

We evaluate a function and pass it two arguments.

We don't do arithmetic in scheme. We just have arithmetic functions.. Literally just + - / and *. They look like prefix arithmetic but they aren't.

I cannot stress enough, c is not the result of actually adding a and b. c is merely defined as (+ a b) which just so happens to be their addition. 

(define c (+ a b))

Scheme has a runtime system that compiles on the fly and executes the code. It is an interpreted language.

As an extra note, we won't be dealing with strings outside of displaying information in our class code. Strings look like so: 

"string" // a string
"a"      // this is all we do for characters

In full scheme, we do have characters and strings. We're limiting it for our class though.

We'll also deal with lists. We can do it like so with the list function: 

(list "a" "b" "c")

There are various important functions that you'll need to know for scheme. Perhaps the most important is the ability to create variables. This is the closest you'll get to assignment in scheme. Remember, there is no memory. As long as the name is alive (there is a scope) the name is bound to the value. That's permanent until you bury it. 

(define name value)

What about a function? How do we define one? In fact, we can define one without a name. To do so, we use a variable called lambda. Lambda will give us a function without a name. Functions are first class function. This is very powerful in production systems. For example, we may have a specific sort we wanna use. Maybe we're just sorting 10 elements. We don't need to merge or quick sort. Just a simple sort will do. So we can pass this along so we don't have to do all that extra setup that comes with those more powerful sorts.

Now then, lambda is a function that takes two arguments. A list of arguments and then an expression. If we see lambda, then the first arg is a list of arguments. Just know that. 

(define name (lambda
  (x y z)(z x y)))

Notice, z is a function! And yet we pass it as a parameter. Remember, in a scheme expression we always have function first followed by args. You may be wondering is x is a function. It's not a full blown statement though. It's a list of arguments in this particular case.

With the previous defined, we can now do this: 

(name 2 3 +)

This evaluates to 5. Our name function just takes x as some value, y as some value, and then z as a function. So, 2 and 3 are our x and y values. + is a function we are passing in.

A scheme list has two parts. A head. And also a tail. So every scheme list looks like this: 

(car (list "a" "b" "c")) // car gives us the list. the head
(cdr (list "a" "b" "c")) // gives us the rest of the list that isn't the head

How do you get the fourth element of the list? Call car four times. A bit yucky but that's how you do it.

Documentation for scheme is at htdg.org. Read chapter 1 of the book. It'll also give you information on racket. Dr. Racket is a simple IDE that'll help you out.

To iterate in scheme, we use recursion. Base case is a bit of nonsense. Only a small subset of recursive programs use base case. For a recursive program, there are three steps involved:

  • Are we there yet?
  • What's the least amount of work
  • Rest of it

A careful look reveals that recursion does not call itself, it calls a copy of itself slightly modified.

We are marching towards a goal with recursion. Recursion is underneath all of the iterative concepts that we've used. For, while, etc loops are all syntactic sugar for recursion. In fact, recursion is one of the most fundamental operations of computing. Arithmetic operations are not. They can all be done by table lookup. Recursion cannot.
