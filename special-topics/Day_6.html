Data Bases

For the next two weeks, we'll be covering data bases.

What is a database? A database contains information. Shows relationships amongst said data (such as with a relational database). In her own words, a database handles storage. A database also helps to structure our data.

DBMS - Databsae Management System. Allows users to read, update, edit, and delete data within a database (remember, CRUD). MySQL, sqlite, oracle, postgres.

Working with databases mostly revolves around CRUD. Create, read, update (edit), delete.

Database Challenges:

  • working with large data
  • security breaches
  • relationships may not be fully clear
  • managing and maintaining a database
  • keeping up to date

Edgar Codd: invented the concept of relational databases (remember this, could be an easy question on the exam).

Relational Database: Stores data in the style of a table. Rows and columns.
Primary Key: uniquely identifies an attribute from a table. For example, name, age, and school could be three different columns. Let's say we want to look up a Steve in the table but we have multiple Steves! So, we have a unique ID to refer to that specific row in which we found Steve.
Record: refers to a row in a relational database table
SQL: Structured Query Language.

Sample Database Problem

Remember, when first presented a problem we always start by breaking it down!

Now then, let's say we were hired to do payroll for all of the employees at some company. Every two weeks, you need to look up who gets paid and give them their checks. For every employee that gets paid, you need to let the manager know. The managers get paid once a month.

Key Info

  • employees paid every 2 weeks
  • alert manager
  • manager gets paid every month

A manager has many employees. An employee belongs to a single manager. This is a one-to-many relationship!

If we were to make a spreadsheet for this, how would we go about it? Something like the following (pretend the table is filled out):

Many-to-many relation. For example, taxis and ubers have many passengers. A ride is a join. Passengers take many ubers/taxis. Taxis/ubers and passengers are connected via the join ride.

When working with relationship databases, it's not just about setting up tables. We need to see the relationships between the various tables/data and combine them via joins.

Joins

On the next exam, she won't be asking too much about joins but it's good to know about them.

Remember, a relational database consists of many tables. To connect those tables, we use joins.

First, we have the inner join. If we had a venn diagram, we'd return where they intersect.
We also have left outer join and right outer join which would return the outer left and right circle in a venn diagram respectively.
A full join would return everything!
A left join and a right join would return the entire left and right circles in their entirety.

Let's examine another relationship such as with Amazon. We have a buyer and a seller.
On Amazon, we have a cart that holds items we wish to buy. That will be joined with a purchase.
An item and a cart don't do anything on their own. However, we can join them together.

We have an item, a cart, and a purchase. Each user has their own cart. A single cart can have many items (items can be purchased mulitple times even).
